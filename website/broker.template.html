02

One session, two channels

The #1 thing people get wrong: the AI agent is not a terminal being screen-scraped. Every session runs two independent things at once. Flip between them.

process
claude -p --output-format stream-json
→
channel
stdin / stdout pipes (no TTY)
→
you get
structured view_event frames

The agent runs headless. The broker spawns it with exec.Command + StdinPipe/StdoutPipe — literally a child process exchanging newline-delimited JSON. Your message is encoded as a stream-json user line and written to its stdin; each assistant/result line it prints becomes a typed chat event pushed to your phone. codex is the same idea but JSON-RPC 2.0 over stdio (a long-lived codex app-server). No terminal, no screen-scraping — a structured machine protocol on a pipe.

process
tmux new-session -A -s conduit-<id>
→
channel
a real PTY (pseudo-terminal)
→
you get
raw bytes + reflowable snapshot

Separately, the session opens a PTY running an interactive login shell inside a per-session tmux. A read loop (drain()) streams the raw bytes to the Terminal tab and feeds a headless VT parser so a fresh client gets a snapshot reflowed to its screen size. Because it's tmux, this shell survives a broker restart — the session state lives outside the broker process, so relaunching it doesn't lose your terminal.

⚠ common misconception

"The broker screen-scrapes the AI's terminal output."  It doesn't. The Chat tab is a headless JSON process; the Terminal tab is the only real PTY, and it runs a shell, not the agent. Only a legacy adapter with no protocol backend would ever be scraped — and nothing shipped uses it.

05

Who can actually reach it

“You started a server on a public VPS — isn’t :1977 open to the whole internet?” No. Two independent walls stand in the way, and an attacker has to beat both. Pick who’s knocking on the box:

wall 1 · network reachability

Bound to loopback, not the world

On a remote box the broker starts with --addr 127.0.0.1:<port>. That single choice is the firewall: 127.0.0.1 is the loopback interface — an address the kernel will only deliver traffic to from the same machine. Bind 0.0.0.0 instead and the port answers the whole internet; bind loopback and a packet arriving on the box’s public IP for :1977 is simply never routed to the listener. Port-scan the VPS and :1977 looks closed. The only listening public port is 22 — SSH — and that’s key-authenticated. So “reach the broker” first requires “already be inside the box.”

wall 2 · application auth

Every call needs the token

Even something already on the box’s loopback can’t drive your sessions. Every HTTP/WS request must carry Authorization: Bearer CONDUIT_TOKEN — a 32-byte URL-safe secret minted at the QR-code pairing and pinned into the systemd unit. The broker compares it in constant time against its in-memory table; no match → 401. (The handshake also accepts ?token= purely because browsers can’t set headers on a WebSocket upgrade.) This is the defense-in-depth half: loopback keeps the internet out; the token keeps everything else on the box out.

so how does the app get in?

It rides your existing SSH trust

The app never dials :1977 across the network — it can’t; nothing can. Instead the Rust core opens an ordinary, authenticated SSH session to the box (the same login you’d use in a terminal) and sets up a local port-forward: a loopback TcpListener on the phone whose every connection is proxied through an encrypted direct-tcpip SSH channel straight into the box’s 127.0.0.1:<port>. The app dials ws://127.0.0.1:<local> on the phone; it surfaces on the box’s loopback. From the broker’s point of view every connection already arrived from localhost — it literally cannot be reached any other way. Compromise means first compromising your SSH login, then still needing the conduit token.

LAN exception: with --local the broker binds a routable interface and advertises over mDNS so a phone on the same trusted Wi-Fi can connect without the tunnel — still behind the same bearer token. That’s a deliberate trade (trusted network instead of SSH), not a hole.

⚠ common misconception

“It’s a server on a public IP, so the port is exposed.”  It isn’t. The broker isn’t on a public interface at all — it sits on loopback behind your SSH login and a bearer token. The public attack surface is exactly your VPS’s SSH daemon, nothing conduit added.

07

Where the code lives

It's a Go monolith under broker/internal/. The two big rooms are ws (the door) and session (the brain). The SSH tunnel isn't even here — it's in the Rust core on the app side.

the other half · core/

Rust core (UniFFI) — and why Rust

The app doesn't open the SSH tunnel in Swift — it calls into a Rust library compiled to an .xcframework (iOS) and an .so (Android), with the Swift/Kotlin API generated by UniFFI. core/src/ssh/tunnel.rs keeps the SSH session alive and forwards a local port into the box's loopback :1977; the broker just listens on localhost.

Why Rust, not Swift + Kotlin twice?

  • Write the hard part once. SSH handshake, the port-forward proxy loop, WebSocket transport, reconnect + the generation-guard state machine — one implementation feeds both apps, so iOS and Android can't drift apart on the gnarly networking.
  • Pure-Rust SSH (russh). No dependency on a system OpenSSH that mobile doesn't ship; the whole tunnel is just library code linked into the app.
  • Memory-safe by construction. This is crypto- and network-facing code parsing untrusted bytes — exactly where C/C++ would invite buffer bugs. Rust gives that safety with no GC pauses.
survival · why Resume works

Restart & recovery

Sessions persist to ~/.conduit/sessions/<id>/ and are recovered lazily — listed on boot, respawned when you reopen them. The tmux server survives a broker restart (KillMode=process reaps only the broker). The agent CLIs are children, so they die — but the transcript is on disk, and a failed send self-heals by respawning + resuming the conversation once.
